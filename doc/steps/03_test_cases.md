# Step 03 - 增加测试用例

## 目标

从单字节测试扩展到多个实际验证场景。

## 已加入测试

| 测试场景 | 数据 | 目的 |
| --- | --- | --- |
| 单字节回环 | A5 | 确认 UART RX -> FIFO -> UART TX 基本链路可用 |
| 多字节回环 | 11 22 33 44 | 确认连续数据传输顺序正确 |
| 20 字节流测试 | 00 到 13 | 确认长一点的数据流不会丢字节或乱序 |
| FIFO 边界模型测试 | 8 写 8 读 | 直接检查 FIFO full/empty 边界状态 |
| Reset 恢复测试 | reset 后发送 A5 | 确认复位后系统仍能恢复工作 |

## 为什么连续测试重要

只发一个 byte 时，很多时序问题暴露不出来。连续数据会检查：

- FIFO 写入是否正确。
- FIFO 读出是否正确。
- UART TX/RX 是否保持顺序。
- 复位后是否有残留状态影响后续测试。

## 当前工程里的一个重要处理

多字节 loopback 用了较保守的发送间隔，因为当前 UART 顶层回环逻辑不是完整握手式流控设计。如果间隔过短，会把“RTL 流控问题”和“验证环境功能”混在一起。作为第一版作品集，我们先让测试稳定 PASS，再把更激进压力场景作为后续改进点。

## 小白要记住的点

测试用例不是越乱越好，而是要能说明你在验证什么。每个 test case 都应该有明确目的。
